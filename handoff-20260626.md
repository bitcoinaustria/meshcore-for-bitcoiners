# MeshCore for Bitcoiners — Presentation Handoff

**Purpose:** Brief for a follow-up session that will build a talk titled *"MeshCore for Bitcoiners."*
**Audience:** Bitcoin community (technically literate, will push back hard on sloppy analogies).
**Tone goal:** Draw genuine structural parallels, but stay intellectually honest — name where each analogy breaks before the audience does.

---

## 1. Core thesis

A keypair replaces institutional identity. Bitcoin replaces the bank account (gatekeeper: the bank). MeshCore replaces the SIM card / phone number / radio call sign (gatekeepers: the telco *and* the state). In both systems you make yourself a full network participant with no registration and no permission.

Extra edge for a Bitcoin crowd: radio spectrum is historically *heavily* licensed and regulated. A permissionless self-issued identity broadcasting in the ether is arguably even more subversive than permissionless money. This is the line the whole talk hangs on.

---

## 2. The identity / address parallel (strongest section)

Both systems are built on a self-generated public/private keypair. Map the details — each with its failure point:

- **Self-generated, permissionless.** Keypair is created on-device, offline; nobody issues it. Holds perfectly. This is the heart.
- **Public key = address, pseudonymous.** Bitcoin address is a *hash* of the pubkey; MeshCore uses a pubkey *prefix* as the routing target. Parallel ("short identifier derived from the pubkey") holds, but motivation differs: Bitcoin hides the pubkey until spend (privacy/quantum), MeshCore's prefix is just a routing shortcut with collision risk.
- **"Not your keys, not your identity."** Lose the key = identity/coins gone, no reset, no central recovery. Culturally identical stance in both communities — rhetorically strong.
- **Out-of-band trust / identical MITM problem.** "Do I really have the right key?" Bitcoin: address-swap malware. MeshCore: verify keys via QR / in person / word lists (TOFU). Same attack surface, same fix.

**Where it breaks (say it out loud):** HD wallets / address rotation. Bitcoin rotates addresses for privacy (one seed → many keys). A MeshCore identity is deliberately *sticky* — you *want* to stay discoverable. No HD equivalent.

---

## 3. The signing model — the elegant twist

This is the subtle bit that makes the talk feel insightful. MeshCore does **not** sign everything:

- **Adverts ARE signed** (full Ed25519 signature over name + position + pubkey) — prevents identity spoofing. This is the "identity anchor."
- **Direct messages are NOT signed** — they're encrypted to the recipient's pubkey (X25519 ECDH → shared secret) with a 2-byte MAC. That proves integrity *to the recipient*, not authorship to the world.
- **Group/channel messages** use a pre-shared key (AES-256-CTR), channel identified by a 1-byte hash of the key. No per-sender signature at all — anyone with the PSK can read *and* write.
  > ✅ **Corrected (verified 2026-06-29):** it is **AES-128, not AES-256**, and **ECB** mode (Encrypt-then-MAC: AES-128-ECB ciphertext + a 2-byte HMAC-SHA256 tag) — *not* CTR. The 1-byte channel id is right (channel id = first byte of `SHA256(key)`). The deck already states AES-128 correctly. Sources: [jacksbrain — *A Hitchhiker's Guide to MeshCore Cryptography*](https://jacksbrain.com/2026/01/a-hitchhiker-s-guide-to-meshcore-cryptography/), [michaelhart/meshcore-decoder](https://github.com/michaelhart/meshcore-decoder), [localmesh.nl — encryption details](https://www.localmesh.nl/en/meshcore-encryption-details/).
- There is an optional signed text type (only when `txt_type == 2`, 4-byte signature). Not the default path.

**The twist to put on a slide:** both systems live in a fully public broadcast medium (blockchain = everyone sees everything; radio = literally transmitted to everyone in range), but their keypairs solve *opposite* problems.

- Bitcoin makes the **content public** and protects the **authorization** → key is for *signing*. A transaction without a valid signature does not exist; the signature *is* the product.
- MeshCore makes the **identity public** (the advert) and protects the **content** → key is for *encryption*. The signature is just the bouncer at the identity door; the actual communication runs on encryption, not signatures.

Same cryptographic material, opposite deployment, because both must cope with "everyone sees the packets" — just at different ends.

## 4. Throughput as a *chosen* constraint

Reframe "low throughput" as deliberate, not a weakness — this lands well with Bitcoiners:

- Bitcoin: block size is a **consensus** cap.
- MeshCore/LoRa: duty-cycle limit in the EU 868 band is a **regulatory** cap (legally limited airtime).

In both cases the throttle is *imposed*, not "they couldn't do more." And that deliberate cap is precisely what forces a second layer to exist. This is the bridge into the Lightning section.

## 5. The Lightning Network section — TWO distinct angles

These are different *kinds* of analogy. Keep them separate on the slides; don't mush them.

### Angle A — Routing logic (the stronger, "actual" parallel)

The mechanism: **advert first → you learn at least one path → you then send along that known path instead of flooding everyone.**

- MeshCore: an advert floods the mesh; as it passes repeaters, path info accumulates. Once a node knows a path, subsequent direct messages are *source-routed* along that specific path rather than re-flooded.
- Lightning: you don't broadcast a payment to the whole network — you find a route through specific channels and send along it.
- **Shared principle:** source-route along a discovered path; don't spam the whole network per message. This is genuinely how *both* keep network-wide load down.

**Where it breaks (be honest):**
- *Discovery differs.* Lightning gossips the full channel topology globally, then computes a path locally (proactive). MeshCore discovers paths reactively, by flooding adverts (AODV-like). Same "use a specific path" outcome, different discovery.
- *No onion privacy.* Lightning onion-routes (hops don't see the full path). MeshCore path bytes are in the clear — repeaters must read the prefixes to forward. No hop privacy.
- *Data, not value.* Lightning routes carry conditional, atomic value (HTLCs). MeshCore routes carry data — no settlement, no atomicity.

### Angle B — Layer separation / the bandwidth jump

The idea: use MeshCore as a constrained "Layer 1" to **bootstrap**, then jump to a "Layer 2" that lives *outside* the limited mesh bandwidth.

- Small mesh message = the channel-open / anchoring event: rare, small, expensive (airtime), but it enables everything after.
- Then: exchange a key / rendezvous over the mesh, and move the bulk transfer to a fat point-to-point pipe (internet, conventional radio, CB, phone) — no longer over MeshCore.
- **Shared principle:** keep the scarce, robust base layer for the few important bytes; push volume off it.

**Where it breaks (the question that *will* come):**
- Lightning permanently *borrows security from L1* — you can unilaterally close a channel and force on-chain settlement. The fast layer stays cryptographically chained to L1.
- The mesh→fat-pipe handoff borrows nothing ongoing. The mesh introduces the parties (key exchange / rendezvous) and then steps away. There is no "settle back onto the mesh," no enforceable fallback.
- So mechanically this is a **one-time bootstrap**, not a continuously anchored channel. Closer to a signaling-plane / media-plane split (WebRTC: slow channel negotiates, media flows direct P2P) or BitTorrent (tracker → direct swarm).

**Recommended framing:** lead with Angle A as the honest "actual" Lightning parallel (it's about routing/topology, which is what LN routing really is). Use Angle B as a forward-looking "what could a real L2 look like?" synthesis — and explicitly label it as a one-time bootstrap, not a trust-anchored channel. That disarms the nerds in the room instead of letting them disarm you.

**Open design question for Angle B (only Harald can answer):** is the single ECDH key from the mesh enough to secure the off-mesh pipe (TOFU), or does it need more? And what's the most compelling pipe to demo — internet (everyday) vs. CB/conventional radio (post-blackout)? That choice sets how "apocalyptic" vs. "practical" the talk feels.

## 6. Verified technical facts (use these; they're checked)

- **Curves:** Ed25519 for signing, X25519 for shared-secret derivation. **Not** secp256k1.
  - Anticipated Q ("why not the same curve as Bitcoin?"): Ed25519 is faster/safer to implement on small MCUs (fewer side-channel pitfalls); Bitcoin is locked to secp256k1 for historical reasons.
- **Identity format:** 32-byte private key + 32-byte public key, persistent. Prefixes `0x00` and `0xFF` are reserved (avoided at generation).
- **Routing prefix:** originally **1 byte** of the pubkey → only 254 usable IDs, so many repeaters share a first byte (collisions). Firmware **1.14+** allows 1-, 2-, or 3-byte adverts/path hashes. This is the collision story.
- **Encryption:** group channels use AES-256 in CTR mode with a PSK; the default public channel uses the well-known weak key (everyone has it) — convenience, not security.
  > ✅ **Corrected (verified 2026-06-29):** group channels use **AES-128 (ECB + 2-byte HMAC-SHA256)**, **not AES-256/CTR** — see the fuller correction in §3. The "well-known weak key on the default public channel" point still holds. Source: [localmesh.nl — encryption details](https://www.localmesh.nl/en/meshcore-encryption-details/).
